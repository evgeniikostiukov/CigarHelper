# .NET / ASP.NET Core (бэкенд CigarHelper)

## Решение

- **CigarHelper.API** — веб-API (minimal hosting `Program.cs`), контроллеры, сервисы, DTO.
- **CigarHelper.Data** — модели, `DbContext`, миграции EF Core.
- **CigarHelper.Import** — отдельный проект импорта; не смешивать ответственность без запроса.

## Слои и зависимости

- API зависит от Data; бизнес-логику по возможности держать в сервисах (`CigarHelper.Api/Services`), не раздувать контроллеры.
- DTO для входа/выхода API — в `CigarHelper.API/Models` (или согласованное с проектом имя); сущности БД — в `CigarHelper.Data/Models`, не отдавать EF-сущности напрямую в публичный API без необходимости.

## ASP.NET Core

- Контроллеры: атрибутный роутинг, явные типы ответов при необходимости; учитывай глобальные настройки в `Program.cs` (например, `SuppressModelStateInvalidFilter`, JSON enum как строки).
- Регистрация сервисов — через `builder.Services` / extension-методы в `Extensions`, если так уже принято в проекте.
- Swagger/OpenAPI и JWT — уже настроены; новые защищённые эндпоинты — через существующие атрибуты авторизации и политики.

## Entity Framework Core

- Контекст и конфигурации — в Data; миграции добавлять стандартными средствами (`dotnet ef migrations add`), не править сырой SQL без причины.
- Запросы: избегай N+1; для списков — проекции в DTO на стороне запроса, где уместно.
- Изменения схемы — только через миграции и согласованное именование.

## C#

- Nullable reference types и стиль проекта — соблюдать; не отключать предупреждения массово.
- Именование: PascalCase для типов/публичных членов; асинхронные методы — суффикс `Async`.
- Исключения: не глотать пустым `catch`; для API — осмысленные HTTP-коды и сообщения, согласованные с фронтом.

## Безопасность

- Не логировать секреты и полные токены; секреты — из конфигурации/переменных окружения, не в коде.
- Проверка прав и ролей — через существующие механизмы (claims, политики), без дублирования логики в каждом методе без нужды.

## Проверка

- После правок: `dotnet build` по решению или затронутым проектам; учитывай анализаторы и стиль IDE.
- Не добавлять пакеты без необходимости; версии — в духе существующих `.csproj`.

## Антипаттерны

- Не переносить бизнес-логику из сервисов в контроллеры «на скорость» без запроса.
- Не менять глобальное поведение JSON/валидации в `Program.cs` без явной задачи.
- Не смешивать слои (например, прямые зависимости Data от API).

## Когда читать дальше

- Нестандартные сценарии (фоновые задачи, интеграции) — смотри существующие модули и повторяй паттерн.
